
摘    要

随着移动互联网的普及和电子商务的快速发展，线上拍卖作为一种特殊的商品交易方式，逐渐受到越来越多用户的青睐。传统的拍卖流程复杂、参与门槛高，而移动端拍卖系统能够让用户随时随地进行商品竞拍，大大降低了参与成本。本系统融合了微信小程序前端、Vue3管理后台和Node.js后端服务，构建了一套完整的移动端拍卖平台。

本系统采用Express.js框架作为后端MVC模式的实现依托，使用TypeScript语言开发，数据库采用PostgreSQL 15配合pgvector向量扩展，缓存层使用Redis 7，实时通信采用WebSocket技术。前端分为两部分：微信小程序原生应用（面向买家和卖家）和Vue3 + Element Plus管理后台（面向拍卖管理员和系统管理员）。系统还集成了AI大模型能力，调用DeepSeek大模型和bge-m3向量嵌入模型，实现了拍品描述智能优化和基于RAG（检索增强生成）的智能估价助手两大AI特色功能。

本文首先通过对移动端拍卖系统的背景和可行性进行分析，展现了该系统的应用前景。然后通过详细的需求分析阐述了本系统的总体结构和需要实现的功能；通过系统设计对系统的总体架构和功能模块进行了规划和设计；通过系统实现的介绍，对该系统具体功能模块的实现方式进行了详细的说明。最后通过系统测试以及系统的运行和使用说明展示了系统的可靠性、健壮性和易操作性。本系统从分析设计到系统实现和测试，均严格遵循软件开发的基本过程。

关键词：拍卖系统；微信小程序；Vue3；Node.js；AI大模型

1	系统需求分析
1.1	系统概述
该系统主要是卖家、买家和管理员进行拍卖交易信息交互的一个系统。卖家可以通过微信小程序发布拍品，设置起拍价和拍卖时间，对拍品进行管理和发货。买家可以通过微信小程序浏览拍卖大厅，参与实时竞价，支付订单和确认收货。拍卖管理员通过Web管理后台审核拍品，受理争议，管理拍卖场次。系统管理员通过Web管理后台管理用户、系统配置和查看操作日志。

系统还集成了AI大模型能力：卖家发布拍品时，可以调用AI拍品描述优化功能，系统根据拍品名称、成色和原价自动生成突出卖点的商品介绍文案；卖家设置起拍价时，可以使用AI智能估价助手，系统通过向量检索同类历史成交记录并结合联网搜索结果，生成估价区间建议和定价策略分析，支持多轮对话。

1.3.3	拍卖与竞价管理
1) 业务概述
拍卖与竞价管理是系统的核心交易环节。卖家对已审核通过的拍品可以开启拍卖，设置拍卖类型（英式递增拍卖/密封投标/荷兰式递减）、起拍价、加价幅度、开始时间和结束时间。系统每10秒执行一次定时任务，自动将到达开始时间的拍卖场次切换为"进行中"状态，自动将到达结束时间的场次进行结算。

买家可以通过WebSocket订阅拍卖场次，实时接收出价更新。出价时系统验证金额必须不低于当前价加加价幅度。出价成功后，系统通过WebSocket广播新价格给所有订阅该场次的用户。同时，Redis保存竞价排行和当前最高价信息。

到达结束时间时，系统自动结算：有最高出价的场次标记为"已成交"，生成待支付订单，写入成交向量到pgvector（用于AI估价）；无出价的场次标记为"已结束"（流拍）。

1.4	系统关键问题分析
本系统开发中，涉及到的关键问题有：
1) 实时竞价通信：拍卖系统要求出价信息能够实时同步给所有参与用户。传统HTTP轮询方式延迟高、资源消耗大。本系统采用WebSocket长连接+发布订阅模式，出价信息通过服务端广播实时推送到所有订阅客户端，确保竞价实时性。

2) 定时任务调度：拍卖场次需要在指定时间自动开拍和自动结拍。系统采用setInterval定时器（10秒间隔）轮询数据库，检查是否有需要开拍或结拍的场次。自动结拍操作涉及多个表的原子性更新（场次状态、出价状态、订单创建），使用Prisma事务（$transaction）保证数据一致性。

3) 向量检索与AI集成：AI估价助手需要从海量历史成交记录中检索与当前拍品相似的记录。系统将成交记录通过bge-m3模型向量化（1024维）存入pgvector，估价时对用户查询进行向量化后使用余弦距离（<=>）检索最相似的5条记录。同时结合Tavily联网搜索获取实时市场行情，构造增强Prompt调用DeepSeek大模型生成估价建议。

4) 微信小程序兼容性：微信开发者工具对ES6+语法支持有限，需要将代码改写为ES5兼容语法（var/function代替const/箭头函数）。同时wx.request默认Content-Type为form-encoded，需要手动设置为application/json。

2	系统概要设计
2.1	系统架构概述
本系统采用前后端分离的B/S架构加微信小程序C/S架构。系统整体分为四层：客户端层（微信小程序 + Vue3管理后台）、服务层（Express.js API + WebSocket）、数据层（PostgreSQL + pgvector + Redis）和AI服务层（DeepSeek大模型 + bge-m3向量模型 + Tavily联网搜索）。

客户端通过HTTP RESTful API与后端进行数据交互，实时竞价通过WebSocket长连接实现。服务层采用JWT进行鉴权，通过中间件实现角色权限控制、操作日志审计和全局错误处理。数据层使用Prisma 7 ORM进行数据库操作，PostgreSQL存储业务数据，pgvector存储向量数据用于AI检索，Redis缓存竞价排行和在线用户信息。

3.4	AI智能估价助手模块
3.4.1	功能说明
AI智能估价助手是本系统的进阶特色功能，基于RAG（检索增强生成）架构。卖家输入商品信息后，系统将查询文本向量化，从pgvector中检索相似的历史成交记录，同时通过Tavily联网搜索获取实时市场行情，将检索结果和搜索结果构造为增强Prompt，调用DeepSeek大模型生成估价区间建议和定价策略分析，并通过SSE流式输出逐字推送到小程序端。系统还支持多轮对话，用户可以追问细节。

5	系统实现
5.1	实现原理
系统采用前后端分离架构。后端采用Express.js框架，使用TypeScript语言开发，遵循RESTful API设计规范。前端分为微信小程序（WXML/WXSS/JS原生）和Vue3 + Element Plus管理后台。

后端通过Prisma 7 ORM操作PostgreSQL数据库，使用@prisma/adapter-pg驱动适配。鉴权采用JWT（jsonwebtoken）方案，密码使用bcryptjs加密。参数校验使用zod库。实时通信使用ws库实现WebSocket服务。

AI能力集成方面，使用OpenAI兼容SDK调用DeepSeek大模型进行文案生成和估价分析，调用bge-m3模型进行文本向量化，调用Tavily Search API进行联网搜索。向量数据存储在pgvector中，通过原生SQL进行余弦距离检索。

系统请求处理流程为：前端发起HTTP请求 → Express路由匹配 → 中间件链处理（CORS/JSON解析/JWT鉴权/角色守卫/审计日志） → 路由处理函数调用Service层 → Service层调用Prisma操作数据库 → 返回统一格式响应。

7) AI智能估价助手：RAG检索能正确找到相似历史成交记录，联网搜索能获取实时行情，大模型生成的估价建议具有参考价值，流式输出体验良好。
